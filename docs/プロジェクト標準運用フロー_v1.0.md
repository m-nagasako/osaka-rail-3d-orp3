# プロジェクト標準運用フロー v1.0

| 項目 | 内容 |
|---|---|
| 版数 | v1.0(OSAKA RAIL 3Dプロジェクトの全工程実績から抽出) |
| 日付 | 2026-07-09 |
| 適用範囲 | 小〜中規模の開発プロジェクト全般。人間+AIの混成チームを想定するが、人間のみのチームでもそのまま使える |
| 使い方 | 新規プロジェクト開始時に本書をdocs/へコピーし、Phase 0から順に実施。各Phaseの完了条件(DoD)を満たすまで次へ進まない |

---

## 全体フロー図

```
Phase 0     Phase 1      Phase 2        Phase 3      Phase 4       Phase 5      Phase 6
立ち上げ → 要件確認 → 外部設計 ─[G1]→ 内部設計 → 実装(縦切り) → テスト(三層) → 納品・公開 ─[G2]→
                          承認ゲート                                                承認ゲート
                                     ┌──────────────────────────────────────┐
                              ─────▶│ Phase 7 運用ループ                      │
                                     │ 課題起票→方針→指示書→実装→独立検証     │
                                     │ →実機確認→クローズ  (以後、繰り返し)   │
                                     └──────────────────────────────────────┘
```

## A. 体制の型(3役)

| 役割 | 責務 | 人でもAIでも可 |
|---|---|---|
| 発注者 | 要件の決定、承認ゲートの判断、実機確認(L3)、公開判断 | 人間必須 |
| PM(方針・文書・管理) | 要件整理、設計、文書体系の維持、課題管理、指示書発行、**成果物の独立検証** | 可 |
| 実装 | 指示書に基づく実装、テスト、作業記録 | 可 |

原則: **PMと実装は分離する**(可能なら別ベンダー・別環境)。実装者の報告を実装者以外が検証できる構造が、品質の最後の砦になる。

## B. 文書体系の型(7文書)

Phase 0で「このPJではどれを使うか」を決める。最低限★は必須。

| 文書 | 役割 | 種別 |
|---|---|---|
| ★外部設計書 | 「何を作るか」の合意。機能・範囲・**対象外**・リスク・マイルストーン | 約束 |
| ★内部設計書 | 「どう作るか」。技術選定(不採用理由込み)・データ設計・テスト計画・トレーサビリティ表 | 作り方 |
| ★QAログ | 起きたこと(作業・障害・設計変更・判断)の時系列。**追記専用・過去を書き換えない** | 記録簿 |
| ★課題管理表 | 未解決の課題・要望・タスクの現在地。状態を更新し続ける | 台帳 |
| 品質管理書 | テスト結果と品質状態の証明。実機チェックリストを含む | 成績表 |
| 障害報告書 | 重大問題の症状/直接原因/根本原因/対処/再発防止/検証 | 事故調査書 |
| 引継ぎ書 | 担当交代時に、文書だけで作業を再開できる状態にする | バトン |

**記録簿(QAログ)と台帳(課題管理表)は別物。**記録だけあって台帳がないと「今なにが開いているか」が見えず、管理不安に直結する(実績PJで発注者から実際に指摘された欠落)。

## C. 横断運用規約(全Phase共通)

1. **ID体系**: D=障害 / R=要望・要件追加 / T=タスク。連番。状態遷移: 未着手→方針決定→指示済→対応中→対応済→クローズ(発注者確認後)
2. **承認ゲート**: G1=外部設計、G2=納品・公開、+スコープ・仕様変更時は都度。**承認をスキップする場合は「誰の指示で・いつ・なぜ」を必ず文書に残す**
3. **独立検証の義務**: 実装者の完了報告は、別環境・別の目で検証してからクローズする(「報告者の環境では本当に成功していた」型の障害は、これでしか見つからない)
4. **正直表示**: 概算・近似・未確定のデータには必ずラベルを付け、出典と年度を明記する。未確定リストは成果物上にも表示する仕組みにする
5. **推測の申告**: 裏取りできていない発言には「推測」と付す。データの実在確認(スパイク)前に計画を確定しない

## Phase 0: 立ち上げ

**やること**: 3役の割当て / 使用する文書の決定(B表から選択) / ID体系と台帳の初期化 / コード置き場(リポジトリ)の決定
**成果物**: 課題管理表(空でよい)、リポジトリ
**DoD**: 「成果物とコードがどこに置かれ、誰が何を承認するか」を全員が言える状態

## Phase 1: 要件確認

**やること**:
- 論点を**3〜5個に絞り、選択肢付きで**発注者に確認する(質問を並べすぎると発注者は考えるのをやめる)
- 実現可能性が怪しい要素は**スパイク(小さな存在確認)**を先にやる。「そのデータは実在するか」「その技術はこの環境で動くか」
- スパイクの結果、不可能と判明したら要件を実現可能な形に翻訳し直して再提示する

**成果物**: 確定要件のメモ、スパイク結果
**DoD**: 発注者の回答が揃い、計画を崩す級の実現性リスクが潰れている
**実績PJでの実例**: 「実ダイヤ再現」の要望に対し、事前調査でダイヤの公開データが存在しないことを確認→「間隔ベースの近似」に要件を翻訳して合意した。設計後に発覚していたら計画ごと崩れていた

## Phase 2: 外部設計 ─ 承認ゲートG1

**やること**:
- 機能・収録範囲・**対象外(作らないもの)**・リスク(対策とセット)・マイルストーンを文書化
- マイルストーンは縦切り(最初のMSで「動くもの」が出る分割)にする
- 提出前に**自己レビュー**を行い、指摘と対処を文書末尾に記録する
- データを扱うPJは、データソースごとに出典・年度・ライセンス・信頼度ラベルを付ける

**成果物**: 外部設計書(自己レビュー記録込み)
**DoD**: **発注者の承認(G1)**
**実績PJでの実例**: 自己レビュー7件のうち1件が重大な考慮漏れ(直通運転)の発見だった。また「対象外」に明記していた区間が後日要望に挙がった際、「障害ではなくスコープ変更」として正しく扱えたのは対象外の明文化のおかげ

## Phase 3: 内部設計

**やること**:
- 技術選定は**採用理由と不採用理由の両方**を書く(後任が同じ検討を繰り返さないため)
- データ構造・主要アルゴリズム・エラー処理を決める
- **テスト計画を設計段階で立てる**。自動化できない領域(実機の見た目等)は、人間の確認工程として正式に計画へ組み込む
- トレーサビリティ表は**機能単位ではなく細目単位**(F5-a、F5-b…)で作る
- 最後に外部設計との**全体整合レビュー**を行う

**成果物**: 内部設計書(自己レビュー+全体整合レビュー記録込み)
**DoD**: 外部設計の全細目に対応が存在し、矛盾ゼロ
**実績PJでの実例**: トレーサビリティを機能単位で取ったため、機能内の細目3件(ランキング表示・視点プリセット・装飾)が実装から漏れた。納品後に発覚し障害D-001〜003となった。**細目単位ルールはこの失敗から生まれた**

## Phase 4: 実装(縦切り)

**やること**:
- 最小範囲(MS1)で全機能を貫通させる。データ整備→中核ロジック→表示→UIの順が基本
- **工程ごとに自己レビューと検証出力を挟む**。「作って進む」ではなく「作って確かめて進む」
- 生成データは必ず「入力→検証→生成物」の一方通行にし、生成物の手編集を禁止する。検証には現実の値との突き合わせ(距離・件数・合計値のサニティチェック)を含める
- 設計とのズレが生じたら、その場でQAログに設計変更として記録する(ズレは悪ではない。記録されないズレが悪)
- データの出典・取得日をパッチファイルのコメントに残す

**成果物**: 動くMS1、更新されたQAログ
**DoD**: 自動テスト全パス+ビルド成功+検証出力の目視確認
**実績PJでの実例**: 検証出力に「3路線以上の乗換駅一覧」を含めていたおかげで、駅の同一判定バグ(別々の駅が1駅に統合)を実装直後に検出できた。検証出力は「動いたか」ではなく「正しいか」が見える形にする

## Phase 5: テスト(三層)

| 層 | 内容 | 実施者 |
|---|---|---|
| L1 単体 | ロジックの自動テスト(純関数化して書く) | 実装者(自動) |
| L2 データ・統合 | データ整合の回帰テスト、ビルド検証、**現実との突き合わせ** | 実装者(自動)+PM |
| L3 実機 | チェックリスト化した手動確認(見た目・操作・体感) | 発注者(手動) |

**やること**:
- L3チェックリストは外部設計の細目から機械的に起こす(細目トレーサビリティの再利用)
- 「テストが通る=正しい」ではない。**現実の既知の値**(実物の所要時間・合計値・順位)との照合を最低1つ入れる
- 修正後の再テストでは、既存テストが全パスすること(回帰)を必ず確認

**成果物**: テスト結果、L3チェックリスト(結果記入欄付き)
**DoD**: L1/L2全パス+L3全項目OK
**実績PJでの実例**: 単体テスト全パスの状態で「所要時間が実物より27%遅い」欠陥があった(パラメータの意味の取り違え)。「実物は約41分のはず」という現実照合だけが検出手段だった。また、単体テストの範囲外(方向反転の座標変換)に潜んだバグはL3で初めて発見された——**三層は互いの死角を埋め合う**

## Phase 6: 納品・公開 ─ 承認ゲートG2

**やること**:
- 品質管理書を作成(L1/L2結果+L3チェックリスト+障害集計+既知の制限)
- **別環境検証**: まっさらな場所に自分でclone/展開し、初期構築から動かす(「作業者の環境では動く」型の障害はこれでしか見つからない)
- 公開前チェック: 機密混入(認証情報・環境変数)がないか / .gitignoreが適切か / 出典・ライセンス表記が生きているか
- コードの置き場を確認する。**「公開サイトがある」と「コードが管理されている」は別物**

**成果物**: 品質管理書、納品物(リポジトリ/zip)
**DoD**: **発注者の承認(G2)**+第三者が再現できる状態
**実績PJでの実例**: ①ローカル作業→直接デプロイでリポジトリ不在となり、コードの唯一の実体がローカルPCだけという状態が発生 ②公開後の独立検証で「新規cloneでは初期構築が失敗する」障害を検出(作業者環境では成功していた)。どちらも本Phaseのチェックで防げる

## Phase 7: 運用・改修ループ

```
課題発生(実機フィードバック・要望・障害)
  → 課題管理表に起票(ID付与・種別判定: 障害/要望/スコープ変更)
  → PMが方針決定(原因分析・優先度・Sprint編成)  ※スコープ変更は承認ゲートへ
  → 作業指示書を発行(下記の型)
  → 実装 → 実装者の完了報告
  → PMの独立検証(clone→構築→テスト→差分レビュー)
  → 発注者の実機確認
  → 課題管理表でクローズ → 次のループへ
```

**種別判定が最初の分かれ道**: 「実装されていない」という指摘は、①障害(設計にあるのに無い) ②新規要望(設計に無い) ③スコープ変更(対象外と合意済み)のどれかで扱いが変わる。外部設計の「対象外」明記がここで効く。

## D. AI協働の型(人間+AI混成チームの追加規約)

1. **AIは「優秀だが記憶が揮発する担当者」**。会話が変われば前提を忘れる。文書がすべてであり、指示書・引継ぎ書の品質が成果を決める
2. **作業指示書の型**: 前提・共通ルール / 原因分析済みの情報 / 実装・修正内容 / テスト追加 / 受入基準 / **着手禁止リスト** / 報告フォーマット。禁止リストは「指示の範囲を広げて解釈する」暴走の防止柵
3. **引継ぎ書の型**: 体制と役割 / 文書マップ(真実の在り処) / 現在の状態 / 運用規約 / 技術申し送り / 禁止事項 / **着任手続き(着任レポート提出)** / アクションキュー
4. **着任試験**: 後任候補に同じ引継ぎ書を渡し、着任レポート(現状要約・課題復唱・矛盾指摘・次の一手)を採点して選定する。観点: 事実の正確さ / 規律準拠 / 推測の申告 / 矛盾検出力 / 提案の質
5. **AIの完了報告も独立検証の対象**。報告が誠実でも、報告者の環境依存の問題は本人には見えない

## E. 失敗カタログ(実績PJで踏んだ地雷と、防止するPhase)

| # | 実際に起きたこと | 防止する仕組み | Phase |
|---|---|---|---|
| 1 | データが存在しない前提で要件が立っていた | スパイク(存在確認)を設計前に | 1 |
| 2 | 直通運転という業務知識の考慮漏れ | 提出前の自己レビューを工程に組込み | 2 |
| 3 | 機能内の細目3件が実装から漏れた | トレーサビリティを細目単位で | 3 |
| 4 | 別々の駅がデータ統合で1駅に潰れた | 検証出力を「正しさが見える」形に | 4 |
| 5 | テスト全パスなのに所要時間が実物と27%ズレ | 現実の既知値との突き合わせをL2に | 5 |
| 6 | 単体テスト範囲外のバグ(逆方向の座標変換) | 三層テストで死角を相互カバー | 5 |
| 7 | リポジトリ不在(コードの実体がローカルのみ) | 置き場確認をG2チェックに | 6 |
| 8 | 新規cloneで初期構築が失敗(ロック不整合) | 別環境での再現検証をG2チェックに | 6 |

## F. 1枚まとめ: Phase別成果物とDoD

| Phase | 成果物 | DoD | ゲート |
|---|---|---|---|
| 0 立ち上げ | 台帳・リポジトリ | 役割と置き場を全員が言える | - |
| 1 要件確認 | 確定要件・スパイク結果 | 実現性リスクが潰れている | - |
| 2 外部設計 | 外部設計書 | 対象外・リスク対策・MS明記 | **G1承認** |
| 3 内部設計 | 内部設計書 | 細目トレーサビリティ矛盾ゼロ | - |
| 4 実装 | 動くMS1 | テスト全パス+検証出力確認 | - |
| 5 テスト | 三層の結果 | L1/L2全パス+L3全項目OK | - |
| 6 納品・公開 | 品質管理書・納品物 | 別環境で第三者が再現可能 | **G2承認** |
| 7 運用 | 台帳・指示書・報告 | 起票→独立検証→クローズが回る | 変更時都度 |
