## 1. 技術スタック選定

| 分類 | 採用 | 理由 |
|---|---|---|
| 言語 | JavaScript (ES Modules) | Nの既知技術。TypeScriptは学習コスト増のため不採用、代わりに主要データ構造へJSDocコメントで型注釈 |
| 3D描画 | Three.js | WebGL 3Dの事実上の標準。npm最新安定版を実装時に固定(package-lockで再現性確保) |
| ビルド | Vite | `npm run dev`で即プレビュー、`build`で静的ファイル出力。Nuxtも内部で使っている道具なので概念が地続き |
| テスト | Vitest | Viteと同系のテストランナー。ロジック層の単体テストに使用 |
| UI | 素のHTML/CSS/DOM | 画面遷移なしの1画面アプリのため、フレームワーク不要 |
| サーバ | なし | 完全静的。`dist/`をNetlify等に置くだけ |

**Nuxt.jsを不採用にした理由**: Nuxtの強みはルーティング・SSR・ページ構成だが、本アプリは「3Dキャンバス1枚+オーバーレイUI」でその全部が余る。素のJSならNのJavaScript知識が直接効き、Three.jsの公式ドキュメント(素のJS前提)ともそのまま対応が取れる。

## 2. アーキテクチャ概要

```
[データ整備層]  scripts/build-data.js   … 元データ → 検証 → public/data/*.json
      ▼ (ビルド時に1回だけ実行)
[アプリ層]
   core/   … シーン初期化・座標変換・状態バス
   sim/    … シミュ時刻・ヘッドウェイ運行計算(純ロジック=単体テスト対象)
   world/  … 3Dオブジェクト(路線・駅・列車・人員バー・地面)
   ui/     … DOMパネル(レイヤ・時刻バー・駅情報・クレジット)
```

状態管理は自作のミニイベントバス(約20行)。`state.set('layers', ...)` すると購読側(world/ui)に通知が飛ぶ。Nuxtで触ったreactiveの「手作り最小版」と思えばいい。単方向: **UI → state → world** で、worldからUIを直接触らない。

## 4. データ設計

### 4.1 座標系

- 緯度経度(WGS84) → ローカル直交座標(メートル)へ等距円筒近似で変換
  - `x = (lng - lng0) × 111320 × cos(lat0)` / `z = -(lat - lat0) × 110540`
  - 原点(lat0, lng0)は**全駅の外接矩形の中心を自動算出**(座標のハードコードを避け、データ差し替えに追従)
- 1 unit = 1m。大阪市スケール(約20km四方)はThree.jsで問題なく扱える
- **縦方向の誇張率 EXAGGERATION = 4(config.jsで可変)**: 駅深度10〜40mは都市スケールでは高低差が見えないため誇張する。UIのクレジットに「深さは4倍に誇張」と明記

### 4.2 stations.json スキーマ

```js
{
  "id": "shinsaibashi",        // ローマ字slug(ユニーク)
  "name": "心斎橋",
  "lat": 34.674, "lng": 135.500,
  "elevM": -20,                // 標高(m)。負=地下、正=高架、0=地上
  "elevSource": "estimate",    // "published"(公表資料) | "estimate"(路線既定値)
  "lines": ["midosuji", "nagahori"],
  "ridership": { "value": 0, "year": "", "source": "" }   // 実装時に投入
}
```

### 4.3 lines.json スキーマ

```js
{
  "id": "midosuji",
  "operator": "Osaka Metro",
  "name": "御堂筋線",
  "color": "#______",          // 公式ラインカラー。実装時に一次資料で確認して投入
  "stations": ["esaka", ..., "nakamozu"],   // 起点→終点の順序が正
  "avgSpeedKmh": 33,           // 表定速度(概算)
  "dwellSec": 25,              // 駅停車秒(概算)
  "service": { "start": "05:00", "end": "24:00" },
  "headways": [                // 時間帯別運転間隔(分)。出典メモ付きで管理
    { "from": "05:00", "to": "07:00", "min": 8 },
    { "from": "07:00", "to": "09:00", "min": 2.5 }
  ]
}
```

- 駅の高低はlines側でなく**駅ごとのelevM**で持ち、線形は駅間を補間する。御堂筋線が中津以北で地上に出る、ニュートラムが高架、といった表現がデータだけで実現できる

### 4.4 meta.json

```js
{ "dataTrack": "bootstrap",    // "bootstrap" | "n02"
  "sources": [ { "name": "...", "license": "...", "year": "..." } ],
  "generatedAt": "...", "notes": ["深さは概算・4倍誇張", "運行は間隔ベースの近似", "直通運転は非対応"] }
```
→ ui/credits.js がこれを読んで出典・注意書きを自動表示する(表記漏れをコードで防ぐ)。

### 4.5 データ整備フロー(二系統)

```
[A系統: 正式データ]  国土数値情報 N02 GeoJSON
   … Nが国交省サイトからDLして scripts/raw/ に配置(この作業環境からは直接DL不可のため)
[B系統: 開発ブートストラップ]  npm "japan-train-data" v0.6.0 (MITライセンス)
   … 本環境で取得可能なことをスパイクで確認済み。ただしデータは2017年時点
        │
        ▼
 scripts/build-data.js ── patches/ をマージ ──▶ 検証 ──▶ public/data/*.json
```

- **スパイク結果(2026-07-07実施)**: `japan-train-data`はnpmから取得可能、パッケージはMIT、バンドル内に日本全国の駅名・座標を内包することを確認済み。元データ系譜はekidata系のため、**公開版ではA系統(N02)への差し替えを推奨**。build-data.jsは両系統を同じ出力形式に変換する(アプリ側は差し替えを意識しない)
- **patches/ の役割**: (1)夢洲駅など2017年以降の開業差分 (2)駅深度 (3)乗降人員(各社公表値) (4)運転間隔 — 出典URLと取得日をコメントで必ず残す
- **検証(build-data.js内 + Vitest)**: 全lines.stationsの参照先が存在 / 緯度経度が大阪近辺(34.3〜34.9, 135.2〜135.7)に収まる / 色がhex形式 / 駅数が期待値と一致 / 隣接駅間距離が0.3〜4km(異常座標の検出) / 梅田〜なんば間の直線距離が約4km前後になる距離サニティ

## 5. 主要アルゴリズム

### 5.1 列車運行 = 「ステートレス時刻関数」方式

列車オブジェクトに状態(現在位置・速度)を持たせない。**シミュ時刻tを入力すると、その瞬間に存在する全列車の位置が計算で出てくる**純関数として実装する。

考え方: 「17:00ちょうど発の列車が17:23にどこにいるか」は、時刻表(出発時刻)と所要時間プロファイルがあれば逆算できる。だから毎フレーム「今この瞬間の全列車」を計算し直す。

```
【前計算】路線×方向ごとに1回だけ
  profile = [ {arr: 到着秒, dep: 出発秒, dist: 起点からの曲線距離m}, ... 駅ごと ]
  (駅間所要 = 曲線距離 ÷ 表定速度、停車 = dwellSec)
  T = 終点到着までの総所要秒
  departures = サービス開始〜終了まで headways に従って出発時刻を列挙

【毎フレーム】シミュ時刻 simT に対して
  for d of departures:
      e = simT - d
      if e < 0 || e > T: continue        # 未出発 or 到着済み → 存在しない
      i = 二分探索(profile, e)
      if e < profile[i].dep:  pos = 駅iに停車中
      else:  r = (e - dep_i) / (arr_i+1 - dep_i)
             pos = 曲線上の距離 lerp(dist_i, dist_i+1, r) の3D座標
```

**この方式の利点**: 倍速・巻き戻し・任意時刻ジャンプが「simTを変えるだけ」で完全に成立し、状態の巻き戻しバグが原理的に起きない。さらに純関数なのでVitestで単体テストできる。

**計算量**: ピーク時の同時列車数は概算で全路線合計150本前後(御堂筋線: 片道所要約50分÷ラッシュ間隔2.5分×2方向≒40本、他路線は間隔が緩い)。150本 × 二分探索(比較5回程度) × 60fps ≒ 4.5万回比較/秒 → 全く問題ない。

### 5.2 描画方式

| 対象 | 方式 | draw call |
|---|---|---|
| 路線(9本) | 駅座標をCatmullRomCurve3で滑らかに補間 → TubeGeometry(半径は誇張、地下区間は発光気味のマテリアル) | 9 |
| 駅マーカー | InstancedMesh(球) ×1。クリック判定はraycastのinstanceId → 駅ID | 1 |
| 列車 | InstancedMesh(箱) ×1、容量256。毎フレーム5.1の結果で行列更新、色は路線色 | 1 |
| 乗降人員バー | InstancedMesh(角柱) ×1。高さ = √(乗降人員) × 係数 | 1 |
| 地面・グリッド・装飾 | 半透明平面+GridHelper+模式的な海岸線ライン(装飾目的・精度非保証と明記) | 〜5 |

合計 約17 draw call。スマホでも余裕を持てる想定(実測はテスト工程で確認)。

- 人員バーを√スケールにする理由: 線形だと梅田(数十万)と小駅(数千)の差が極端になり画が壊れる。√で圧縮しつつ、実数値は駅情報パネルに表示して誤解を防ぐ。この変換はUI上にも明記する
- **線形(路線の形)は駅間補間による近似**であり実際の線路形状ではない。クレジットに明記

### 5.3 レイヤ連動規則

- 路線ON/OFF → 路線チューブ・その路線の列車・人員バーが連動
- **乗換駅は「所属する全路線がOFFのときだけ」非表示**(片方OFFで消えると不自然なため)
- 地下/地上・高架フィルタは駅のelevMと線分の高さで判定
- ソロボタン = 対象路線以外を一括OFF(トグルで復帰)

## 6. UI実装方針

- パネルは素のDOM。3Dキャンバスの上にabsolute配置
- レイヤパネル: 事業者見出し(M1はOsaka Metroのみ) > 路線チェックボックス+色チップ+ソロボタン
- 時刻バー: スライダー(05:00〜24:30) + 再生/停止 + 倍速プリセット {×1, ×10, ×60, ×300} + デジタル時計表示
- 駅情報パネル: 駅クリックで表示。駅名 / 路線 / 乗降人員(年度・出典) / 深さ(概算ラベル) / ピン留めボタン(最大5駅の比較リスト)
- モバイル: パネルは下からのシート化。`renderer.setPixelRatio(min(devicePixelRatio, 2))`で高DPI端末の負荷を抑制。OrbitControlsのタッチ操作(1本指回転・2本指ズーム/パン)を有効化

## 7. エラー処理・フォールバック

| 事象 | 挙動 |
|---|---|
| data/*.json の読込失敗 | 画面中央にエラーメッセージ(再読込ボタン付き)。白画面で黙らない |
| データ検証エラー(参照切れ等) | build-data.jsの時点で失敗させる=壊れたデータは配布物に入らない |
| WebGL非対応環境 | 検出して案内文を表示 |
| 時刻がサービス時間外 | 列車0本を正常descriptionとして扱う(深夜は走ってないのが正しい) |

## 8. テスト計画(三層)

**制約の明示**: この作業環境にはヘッドレスブラウザを導入できない(Chromium取得先が許可ドメイン外)。よって描画層の自動テストは不可能で、以下の三層で品質を担保する。

| 層 | 内容 | 実施者 |
|---|---|---|
| L1 単体テスト(Vitest) | projection(往復変換・既知距離)、schedule(列車数がヘッドウェイと整合/位置の単調増加/停車時間/サービス時間外0本)、build-dataの検証関数 | Claude(自動) |
| L2 ビルド・データ検証 | build-data.js実行成功、生成JSONのスキーマ・整合検証、`vite build`成功、distサイズ確認 | Claude(自動) |
| L3 手動チェックリスト | 実機ブラウザでの描画・操作確認(起動/レイヤ/ソロ/時刻ジャンプ/倍速/駅クリック/バー切替/スマホ操作)。項目表は品質管理書に添付 | N(実機) |

品質管理書には L1/L2の実行結果(件数・パス状況)と L3のチェックリスト(N記入欄付き)を記載する。
