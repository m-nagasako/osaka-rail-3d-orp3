## 2. 原因分析

- **直接原因**: 工程③⑤の実装時に、外部設計の機能記述のうち括弧内・列挙途中の細目(プリセット視点、ランキング、装飾ライン)を実装対象から取りこぼした
- **根本原因**: 内部設計書11.1の機能トレーサビリティ確認が「担当モジュールが存在するか」の粒度で行われており、**機能内の細目単位での照合をしていなかった**。モジュールが存在すれば「対応済」と判定される確認方法だったため、細目の欠落を検出できない構造だった

## 3. 再発防止策

- M2着手時、トレーサビリティ表を「機能ID」単位から**「機能ID-細目番号」単位**(例: F5-a/F5-b/F5-c)に再作成し、実装完了時のセルフチェックを細目単位で行う
- 品質管理書のL3チェックリストを外部設計の細目から機械的に起こす運用にする(今回のD-001〜003もL3に#19〜21として追加済み)

## 4. 承認プロセスに関する特記事項
